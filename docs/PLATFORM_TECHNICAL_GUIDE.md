# Production ERP Builder — Техническое описание и инструкция по эксплуатации

> Документ описывает текущую реализацию метаданных-конструктора (Production ERP Builder).
> Предназначен для передачи ИИ-ассистенту, который будет формировать задачи на **настройку и наполнение системы реальными данными**.
> Всё в платформе — это строки в PostgreSQL: страницы, сущности, поля, статусы, роли, переводы, представления, виджеты. Исходный код менять не требуется — конфигурация делается через интерфейс администратора и/или REST API.

---

## 0. Архитектура в двух абзацах

- **Метаданные-driven.** Нет «захардкоженных» экранов и маршрутов. Боковое меню целиком строится из таблицы `pages`. Структура данных (сущности, поля, статусы, связи) — это записи в БД.
- **Contract-first.** Каждый эндпоинт описан в `lib/api-spec/openapi.yaml`; из него генерируются клиентские хуки (React Query) и Zod-схемы. Сервер валидирует вход теми же Zod-схемами. Контракт — единственный источник правды для API.
- **Стек:** pnpm-монорепо, Node.js 24 / TypeScript, Express 5 (API, порт 5000), PostgreSQL + Drizzle ORM, React + Vite (фронтенд `erp-platform`).
- **Мультиязычность.** Контент хранится как JSONB `{ru,en,he}` на каждой строке и отображается через хелпер `getML()` с цепочкой fallback ru → en → he.
- **Авторизация.** Кастомный email/password + JWT. Самостоятельной регистрации нет — пользователей создаёт администратор. Права (RBAC) читаются из БД **на каждый запрос** (не зашиты в JWT), поэтому изменение роли действует со следующего запроса.

## 1. Существующие сущности (системные таблицы)

Это «системные» сущности платформы (таблицы БД, источник `lib/db/src/schema/*`). Их **не создаёт** пользователь — они есть всегда. Пользовательские (бизнес-)сущности создаются поверх них через конструктор (см. раздел 10).

| Таблица | Назначение |
|---|---|
| `users` | Учётные записи (email, хэш пароля, имя, `roleId`, язык, направление текста, стартовая страница, активность). Пароль может быть `null` — это «беспарольный» (гостевой) пользователь. |
| `roles` | Роли с структурированным `permissionsJson` (RBAC). |
| `pages` | Страницы и пункты меню (источник бокового меню). Поддерживают вложенность (`parentPageId`), зеркальные страницы (`mirrorEntityId`) и тип «дашборд» (`isDashboard`). |
| `entities` | Бизнес-сущности (таблицы данных): системный ключ, мультиязычное имя, иконка, привязка к странице (`pageId`). |
| `entity_fields` | Поля (колонки) сущностей. |
| `entity_statuses` | Статусы жизненного цикла записей сущности (цвет, default/final/archive-trigger). |
| `entity_transitions` | Переходы статусов = «workflow» сущности. |
| `entity_records` | Сами данные (записи). Значения — JSONB-карта `fieldKey → value`. |
| `relations` | Типизированные связи между сущностями. |
| `record_links` | Конкретные связи между записями (с денормализованным типом для контроля кардинальности). |
| `views` | Сохранённые представления (фильтры/сортировки/поиск) на сущность. |
| `page_fields` | Поля, локальные для страницы (доп. колонки на зеркальной странице, не трогающие схему исходной сущности). |
| `page_record_values` | Значения page-local полей, отдельно от данных сущности. |
| `dashboard_widgets` | Виджеты дашборда (метрики, графики, таблицы, заметки). |
| `translations` | Строки интерфейса (ключ → `{ru,en,he}`). |
| `modules` | Реестр подключаемых модулей (например, `google_drive`). |
| `app_settings` | Синглтон-строка брендинга (имя приложения, подзаголовок, логотип, символ валюты). |
| `system_events` | Журнал внутренних событий (best-effort). |
| `audit_log` | История изменений записей (кто/когда/что, old→new). |
| `login_history` | История входов (IP, user-agent). |
| `guest_links` | Гостевые ссылки (хэш токена, срок, отзыв). |
| `deleted_files` | Корзина для локальных файлов, удалённых из записей. |
| `google_drive_connection` / `google_drive_folders` | Подключение Google Drive и управляемые папки загрузки. |

## 2. Поддерживаемые типы полей

Источник правды — enum `FieldType` в OpenAPI; серверная валидация значений — `validateValues()` в `artifacts/api-server/src/routes/records.ts`.

| Тип | Хранимое значение | Валидация на сервере |
|---|---|---|
| `text` | строка | строка |
| `textarea` | строка (многострочный) | строка |
| `number` | число | конечное число (`Number.isFinite`) |
| `boolean` | true/false | boolean |
| `date` | строка-дата | парсится `Date.parse` |
| `datetime` | строка дата-время | парсится `Date.parse` |
| `select` | строка | значение должно быть из `optionsJson` |
| `email` | строка | regex e-mail |
| `url` | строка | валидный URL, **только http/https** |
| `phone` | строка | regex телефона |
| `user` | id пользователя (число) | целое > 0, **проверяется существование пользователя**; можно ограничить выбор ролями (`userConfigJson.allowedRoleIds`) |
| `file` | полиморфный объект (см. ниже) | по `kind` + разрешённым источникам |
| `function` | **не хранится** | вычисляется при чтении из формулы `formulaConfigJson.expression` (ссылки на поля `{field_key}`), безопасный песочница-эвалуатор |
| `relation` | **page-field только** | колонка, подтягивающая одно поле связанной записи через одиночную связь (значение производное, правки пишутся в связанную запись) |

**Поле `file` (мультиисточник).** Значение полиморфно по `kind`:
- `server` — загрузка в объектное хранилище: `{kind:"server", path, name, contentType?, size?}`
- `gdrive` — загрузка в управляемый Google Drive (хранится `fileId`, не байты): `{kind:"gdrive", fileId, name, ...}`
- `link` — вставленная внешняя ссылка: `{kind:"link", url, name?}`
- Легаси-значение без `kind`, но с `path` → трактуется как `server`.
- Админ выбирает разрешённые источники на поле через `fileConfigJson.allowedSources` (по умолчанию — только `server`).

**Доп. настройки поля (JSONB-колонки `entity_fields`):**
- `isRequired`, `defaultValue`, `optionsJson` (для `select`).
- `permissionsJson` — права по ролям (hidden/view/edit).
- `isFilterable` — поле доступно в панели фильтров.
- `showInTable` — показывать колонку в таблице.
- `showColumnTotal` + `totalFillColor`/`totalTextColor` — сумма по числовой колонке (итоговая строка).
- `formatRulesJson` — условное форматирование (правила окраски ячейки/строки/текста).
- `sortOrder` — порядок колонок.

## 3. Поддерживаемые типы связей

Источник: `relations.ts`. Enum `RELATION_TYPES`:

| Тип | Кардинальность |
|---|---|
| `one_to_one` | 1:1 |
| `one_to_many` | 1:N |
| `many_to_one` | N:1 |
| `many_to_many` | N:N |

- Связь определяется между `sourceEntityId` и `targetEntityId`, имеет ключ, прямое и обратное имя (мультиязычные).
- Конкретные связи записей хранятся в `record_links`. **Кардинальность обеспечивается на уровне БД** частичными уникальными индексами по денормализованному `relationType` (поэтому тип копируется в строку `record_links`).
- Тип связи **нельзя менять, пока есть связанные записи**; смена типа и вставка связи сериализуются блокировкой строки (`SELECT … FOR UPDATE`), чтобы денормализованный тип не «разъехался» с родителем.
- На странице можно показать одно поле связанной записи (page relation field) и метрики по связям (агрегируются **на связь**, а не на уникальную запись).

## 4. Права доступа (RBAC)

Источник: `roles.ts` (`RolePermissions`) + memory `rbac-permissions.md`, `erp-field-row-permissions.md`.

Структура `permissionsJson` роли:
```jsonc
{
  "superAdmin": false,              // обходит ВСЕ проверки (сеяная роль admin id=1)
  "admin": {                        // доступ к админ-конструкторам
    "pages": false, "entities": false, "roles": false, "users": false,
    "translations": false, "events": false, "modules": false,
    "googleDrive": false, "settings": false
  },
  "pageIds": [],                    // какие контент-страницы видны/доступны
  "records": {                      // права CRUD по сущностям, ключ = entityId (строкой)
    "12": {
      "view": true, "create": true, "update": true, "delete": false,
      "scope": "own",              // "all" (все строки) | "own" (только свои)
      "scopeFieldKeys": ["manager"] // поля типа user, определяющие владельца
    }
  }
}
```

Ключевые правила (одинаковы на сервере и в UI, но **сервер — единственная жёсткая граница**, UI лишь косметика):
- `superAdmin` обходит всё.
- `admin.entities` управляет **всем семейством** конструктора сущностей (сущности + поля + статусы + связи + представления) — одна способность, не по отдельным под-конструкторам.
- Запись CRUD — **по каждой сущности**; **write подразумевает view** (нельзя дать create/update без view).
- **Поля (field-level):** `permissionsJson` поля = `roleId → hidden|view|edit`. Если явной записи нет — наследуется от прав записи: `create || update ⇒ edit`, иначе `view`. Скрытые поля **не наблюдаемы** — вырезаются из всех ответов (включая ответы create/update) и исключаются из whitelisting фильтров/поиска.
- **Строки (row-level):** `scope: own` показывает строки, где любое из `scopeFieldKeys` (поля типа user) равно id текущего пользователя. `own` + пустой `scopeFieldKeys` ⇒ **ноль строк** (так задумано). Для чужих строк GET/UPDATE/DELETE возвращают **404** (не 403), чтобы не раскрывать существование.
- **Merge-on-write:** при create/update берутся только поля, которые роль может `edit`; остальные сохраняют прежнее значение (update) или отбрасываются (create).
- Права читаются **свежими из БД на каждый запрос**.
- Метаданные (GET страниц/сущностей/полей/статусов/связей/представлений/дашборда) — только auth (чтобы оболочка отрисовалась); сами данные — по правам.

**Гостевой доступ:** «клиент = беспарольный пользователь». Админ создаёт беспарольную учётку + роль «Гость», генерирует неугадываемую ссылку, которая выдаёт read-only JWT. Read-only — жёсткий guard на уровне `requireAuth` (а не RBAC); RBAC при этом управляет меню/колонками/«своими» строками.

**Имперсонизация:** админ может «войти как пользователь» поверх кастомного JWT с защитой от эскалации.

## 5. Представления (Views)

Источник: `views.ts` + memory `record-views-query.md`.

- Представление привязано к сущности (`entityId`), имеет ключ, мультиязычное имя и `configJson`.
- Структура `configJson`:
  ```jsonc
  {
    "filters": [ { "field": "status", "operator": "equals", "value": "open" } ],
    "filterConjunction": "and",   // and | or
    "sorts":   [ { "field": "createdAt", "direction": "desc" } ],
    "search":  "строка поиска",
    "visibleFields": []           // СЕЙЧАС не влияет на колонки (см. ниже)
  }
  ```
- Запрос данных — **POST `/entities/{entityId}/records/query`** (а не GET): богатое дерево фильтров не помещается в query-string, и POST устраняет коллизию кодогенерации. Ключи полей в фильтрах/сортировках валидируются по whitelist активных **видимых** полей (400 на неизвестное); значения — связанные параметры (защита от инъекций).
- **Важно:** видимость колонок задаётся **НЕ представлением**, а флагом `showInTable` каждого поля (+ права роли на поле). Представление несёт только фильтр/сортировку/поиск. `visibleFields` остаётся в типе ради обратной совместимости, но на отображение не влияет.
- На сущности может быть не более одного представления по умолчанию (`isDefault`, частичный уникальный индекс).

**Панель фильтров (зависимые фильтры):** поиск + быстрый фильтр по статусу + опциональные выпадающие фильтры по полям с `isFilterable`; списки опций сужаются по другим активным фильтрам.

## 6. Workflow (переходы статусов)

Источник: `transitions.ts` + memory `workflow-engine.md`.

- Отдельной таблицы workflow нет. **Workflow сущности = набор её `entity_transitions`** поверх `entity_statuses`.
- Переход: `fromStatusId → toStatusId` с полями:
  - `allowedRoleIds` — какие роли могут выполнить (пусто = все роли).
  - `requiredFieldKeys` — поля, которые должны быть непустыми для перехода.
  - `actionsJson` — действия при переходе; сейчас поддерживается **только** `{type:"set_field", fieldKey, value}`.
- `fromStatusId` **nullable**: `null` = «из любого статуса» (wildcard-переход в `toStatus`).
- **Приоритет разрешения:** сначала ищется конкретный переход `from→to`, и только если его нет — wildcard. Wildcard не должен «затенять» конкретное ребро.
- **Граница (сервер — жёсткая, клиент — косметика):**
  - Сущность **без переходов** → свободная смена статуса (обратная совместимость).
  - Запись с `null`-статусом не управляется workflow.
  - `superAdmin` обходит весь workflow (включая действия).
  - Иначе смена статуса должна совпасть с определённым переходом (иначе 422), пройти `allowedRoleIds` (иначе 403), применить `set_field` действия **до** финальной валидации и иметь непустые `requiredFieldKeys`.
- **Конкурентность:** финальный UPDATE статуса — это compare-and-set (предикат проверяет, что статус строки всё ещё равен тому, против которого валидировали); 0 строк → **409** (повторить).

## 7. Архивирование

Источник: `records.ts` (поля `archivedAt`, `statusChangedAt`, `archiveExempt`) + memory `archive-engine.md`.

- Архив — это **флаг** на записи (`archivedAt`), не отдельная таблица.
- Статус можно пометить `isArchiveTrigger` с `archiveAfterDays`.
- **Авто-архивация** — ленивый «sweep» при каждом чтении списка/запроса: проставляет `archivedAt` строкам, «висящим» в триггер-статусе дольше `COALESCE(statusChangedAt, createdAt) + archiveAfterDays`. При `archiveAfterDays = 0` смена в триггер-статус архивирует сразу.
- `statusChangedAt` — базовая точка отсчёта; ставится при создании и при каждой смене статуса.
- **`archiveExempt`** (внутреннее булево, не в API): ручное разархивирование (`/records/{id}/unarchive`) выставляет его, чтобы следующий sweep не заархивировал запись снова; снимается при следующей смене статуса. Разархивирование всегда **явное**; уход из триггер-статуса сам по себе не разархивирует.
- Ручные `/records/{id}/archive` и `/records/{id}/unarchive` доступны явно.

## 8. Ограничения (текущие)

- **Самостоятельная регистрация отсутствует** — пользователей заводит только администратор.
- **Workflow-действия** ограничены типом `set_field`. Нет действий «отправить письмо / уведомление / вызвать webhook».
- **Система событий** (`system_events`) — это **только журнал** (best-effort), без движка автоматизаций/обработчиков. Это фундамент для будущих автоматизаций, но самих автоматизаций ещё нет.
- **Формулы (`function`-поля)** работают **в пределах одной записи** (ссылки на поля той же записи). Кросс-записевых вычислений нет, кроме page-метрик по связям.
- **Конкурентное редактирование заметок-виджетов** — last-write-wins (нет версионирования при правке содержимого заметок).
- **Представления не управляют набором колонок** (это решает `showInTable` на поле).
- **Массовый импорт/экспорт записей** (CSV/Excel) в API отсутствует — данные заводятся через record-эндпоинты по одной/через UI.
- **Версионирование файлов** отсутствует; есть лишь корзина локальных файлов (`deleted_files`). Файлы Google Drive и вставленные ссылки в корзине не отслеживаются и не удаляются.
- **Зеркальная страница / дашборд / привязка к сущности** взаимоисключающи для одной страницы.
- На сущность — **не более одного статуса по умолчанию** и **одного представления по умолчанию** (частичные уникальные индексы).
- Одна сущность — одна привязанная страница (enforced на сервере).

## 9. Чего пока нет (направления развития)

- Движок автоматизаций поверх событий (триггеры → действия, расписания).
- Расширенные workflow-действия (уведомления, e-mail, webhooks, вызовы внешних API).
- Rollup/lookup-поля как полноценный тип (агрегации связанных записей в колонке сущности).
- Импорт/экспорт данных (CSV/Excel), массовые операции над записями.
- Версионирование/история файлов.
- Compare-and-set (оптимистическая блокировка) для контента заметок.
- Полнотекстовый поиск/индексация за пределами текущего поиска по полям.

## 10. Что можно создавать через интерфейс

Через админ-конструктор (страницы под `artifacts/erp-platform/src/pages/admin/`) пользователь с соответствующими правами создаёт:

| Объект | Экран / способность | Способность RBAC |
|---|---|---|
| Страницы и вложенные группы меню | `/admin/pages` | `admin.pages` |
| Сущности (таблицы данных) | `/admin/entities` | `admin.entities` |
| Поля сущности | конструктор полей / setup-mode таблицы | `admin.entities` |
| Статусы сущности | конструктор статусов | `admin.entities` |
| Связи между сущностями | конструктор связей | `admin.entities` |
| Представления | конструктор представлений | `admin.entities` |
| Workflow (переходы) | конструктор workflow | `admin.entities` |
| Записи (данные) | таблица записей (инлайн-редактирование) | права `records` по сущности |
| Роли и права | `/admin/roles` | `admin.roles` |
| Пользователи (+блок/разблок, сброс пароля, гостевые ссылки) | `/admin/users` | `admin.users` |
| Переводы интерфейса | `/admin/translations` | `admin.translations` |
| Модули (вкл/выкл) | `/admin/modules` | `admin.modules` |
| Виджеты дашборда | inline edit-mode на странице-дашборде | `admin.pages` |
| Брендинг / общие настройки | `/settings` | `admin.settings` |
| Корзина файлов | `/admin/file-trash` | только superAdmin |

**Системный ключ при создании.** Везде, где требуется «Системный ключ» (сущность, поле, статус, представление, связь, модуль), его можно **оставить пустым** — он сгенерируется автоматически из названия (транслитерация кириллицы) с проверкой уникальности (при коллизии добавляется `_2`, `_3`, …). Формат ключа: строчные латинские буквы, цифры и подчёркивания, начинается с буквы.

---

## 11. Структура метаданных

«Метаданные» — это конфигурация системы (всё, что не пользовательские данные-записи). Хранятся как строки:

- **Мультиязычные имена/описания** — JSONB `{ru,en,he}` (`nameJson`, `descriptionJson`, и т.п.). Отображение через `getML()` (fallback ru → en → he).
- **Страница (`pages`):** `nameJson`, `icon`, `path` (на нём строится маршрут и меню), `parentPageId` (вложенность), `mirrorEntityId` + `mirrorFieldKeysJson` (зеркальная страница), `isDashboard` (страница-дашборд), `widgetsCollapsedDefault`, `sortOrder`, `isActive`.
- **Сущность (`entities`):** `entityKey` (uniq), `nameJson`, `icon`, `pageId` (FK → `pages`, `ON DELETE SET NULL`), `sortOrder`, `isActive`.
- **Поле (`entity_fields`):** `entityId` (FK cascade), `fieldKey` (uniq в рамках сущности), `fieldType`, `isRequired`, `defaultValue`, `optionsJson`, `permissionsJson`, `fileConfigJson`, `userConfigJson`, `formatRulesJson`, `formulaConfigJson`, флаги отображения (`isFilterable`, `showInTable`, `showColumnTotal`, `total*Color`), `sortOrder`.
- **Статус (`entity_statuses`):** `statusKey`, `nameJson`, `color`, `isDefault`, `isFinal`, `isArchiveTrigger`, `archiveAfterDays`, `sortOrder`.
- **Связь (`relations`):** `sourceEntityId`, `targetEntityId`, `relationKey`, `relationType`, `nameJson`, `inverseNameJson`, `settingsJson`.

Целостность поддерживается реально (в БД/API), а не подразумевается: FK с каскадами/SET NULL, уникальные и частичные уникальные индексы, серверные проверки (одна сущность на страницу, один статус/представление по умолчанию, взаимоисключающие типы страниц).

---

## 12. Структура хранения записей

- **Запись (`entity_records`):** `entityId` (FK cascade), `valuesJson` (JSONB-карта `fieldKey → value`), `statusId` (FK → `entity_statuses`, `ON DELETE SET NULL`), `archivedAt`, `statusChangedAt`, `archiveExempt` (внутр.), `createdAt`, `updatedAt`.
- Значения **типобезопасно валидируются на сервере** по типу каждого поля (`validateValues()`); неизвестные ключи отклоняются (никаких «мусорных» колонок).
- `function`-поля **не хранятся** — вычисляются при чтении.
- **Page-local поля:** значения хранятся отдельно в `page_record_values` (`pageId` + `recordId` → `valuesJson`), чтобы доп. колонки зеркальной страницы не загрязняли данные исходной сущности.

---

## 13. Структура хранения прав

- Права хранятся **на роли** (`roles.permissionsJson`, JSONB), структура — см. раздел 4.
- Пользователь ссылается на роль (`users.roleId`); права читаются свежими из БД на каждый запрос.
- Поле-уровневые права — в `entity_fields.permissionsJson` (`roleId → hidden|view|edit`).
- Строко-уровневый scope — внутри `records[entityId]` роли (`scope` + `scopeFieldKeys`).
- Доступ к контент-страницам — массив `pageIds` роли.

---

## 14. Структура хранения workflow

- Нет отдельной таблицы «workflow». Хранится как набор **переходов** в `entity_transitions`:
  - `entityId`, `fromStatusId` (nullable = «из любого»), `toStatusId`, `nameJson`, `allowedRoleIds` (JSONB array), `requiredFieldKeys` (JSONB array), `actionsJson` (JSONB array действий `set_field`), `sortOrder`.
- Уникальность: частичный индекс на конкретные рёбра `(entityId, fromStatusId, toStatusId)` где `fromStatusId is not null`; не более одного wildcard на целевой статус `(entityId, toStatusId)` где `fromStatusId is null`.
- Сами статусы — в `entity_statuses` (см. разделы 7 и 11).

---

## 15. Структура хранения переводов

- Таблица `translations`: `translationKey` (uniq) → `translationsJson` (`{ru,en,he}`).
- Нет локальных json/yaml-файлов — все строки UI в БД. Веб читает через `useListTranslations`, резолвит `useT(key, default)`; метаданные (мультиязычный JSONB) — через `useML()`.
- Динамические/template-ключи (`t(\`fields.type.${x}\`, …)` и т.п.) необходимо добавлять в `scripts/src/data/ui-translations.json` и пере-сеять командой `pnpm --filter @workspace/scripts run seed-translations` (сеяние идемпотентно).
- Страница логина по дизайну на RU (запрос переводов включается только после входа).

---

## 16. Структура хранения представлений

- Таблица `views`: `entityId` (FK cascade), `viewKey` (uniq в рамках сущности), `nameJson`, `configJson` (структура — см. раздел 5), `isDefault` (≤1 на сущность), `sortOrder`, `isActive`.

---

## 17. Примеры создания (через REST API)

> Базовый префикс API — `/api` (через общий прокси). Все примеры требуют заголовок авторизации (JWT), кроме логина. Имена полей и тела соответствуют OpenAPI-контракту.
> `nameJson` всегда мультиязычный объект. `Системный ключ` можно не передавать/передать пустым — сгенерируется автоматически.

### 17.1. Создание новой страницы
```http
POST /api/pages
Content-Type: application/json

{
  "nameJson": { "ru": "Клиенты", "en": "Customers", "he": "לקוחות" },
  "icon": "users",
  "path": "/customers",
  "parentPageId": null,
  "sortOrder": 10,
  "isActive": true
}
```
- Страница-дашборд: добавить `"isDashboard": true` (нельзя одновременно с привязкой сущности и `mirrorEntityId`).
- Зеркальная страница: `"mirrorEntityId": <id сущности>` (+ опц. `"mirrorFieldKeysJson": ["name","phone"]`).
- Порядок/вложенность меняются через `POST /api/pages/reorder`.

### 17.2. Создание новой сущности и привязка к странице
```http
POST /api/entities
Content-Type: application/json

{
  "entityKey": "",
  "nameJson": { "ru": "Клиент", "en": "Customer", "he": "לקוח" },
  "icon": "user",
  "pageId": 42,
  "sortOrder": 0
}
```
(`entityKey: ""` → сгенерируется автоматически, например `customer`.)

### 17.3. Создание нового поля
```http
POST /api/entities/{entityId}/fields
Content-Type: application/json

{
  "fieldKey": "",
  "nameJson": { "ru": "Телефон", "en": "Phone", "he": "טלפון" },
  "fieldType": "phone",
  "isRequired": true,
  "isFilterable": true,
  "showInTable": true,
  "sortOrder": 20
}
```
Примеры по типам:
- `select`: `"fieldType": "select", "optionsJson": ["A","B","C"]`
- `user` с ограничением по ролям: `"fieldType": "user", "userConfigJson": { "allowedRoleIds": [3,4] }`
- `file` с источниками: `"fieldType": "file", "fileConfigJson": { "allowedSources": ["server","gdrive","link"] }`
- `function` (вычисляемое): `"fieldType": "function", "formulaConfigJson": { "expression": "{price} * {qty}" }`
- числовой итог: `"fieldType": "number", "showColumnTotal": true`

### 17.4. Создание статуса (для workflow)
```http
POST /api/entities/{entityId}/statuses
Content-Type: application/json

{
  "statusKey": "",
  "nameJson": { "ru": "В работе", "en": "In progress", "he": "בעבודה" },
  "color": "#2563eb",
  "isDefault": true,
  "isArchiveTrigger": false,
  "sortOrder": 10
}
```
Статус авто-архивации: `"isFinal": true, "isArchiveTrigger": true, "archiveAfterDays": 30`.

### 17.5. Создание workflow (перехода между статусами)
```http
POST /api/entities/{entityId}/transitions
Content-Type: application/json

{
  "fromStatusId": 11,          // null = "из любого статуса" (wildcard)
  "toStatusId": 12,
  "nameJson": { "ru": "Взять в работу", "en": "Start", "he": "התחל" },
  "allowedRoleIds": [3],        // [] = разрешено всем ролям
  "requiredFieldKeys": ["assignee"],
  "actionsJson": [
    { "type": "set_field", "fieldKey": "startedAt", "value": "2026-01-01" }
  ],
  "sortOrder": 0
}
```
Как только у сущности появляется хотя бы один переход, смена статуса её записей ограничивается определёнными переходами (для не-superAdmin).

### 17.6. Создание представления
```http
POST /api/entities/{entityId}/views
Content-Type: application/json

{
  "viewKey": "",
  "nameJson": { "ru": "Открытые", "en": "Open", "he": "פתוחים" },
  "isDefault": true,
  "configJson": {
    "filters": [ { "field": "stage", "operator": "equals", "value": "open" } ],
    "filterConjunction": "and",
    "sorts": [ { "field": "createdAt", "direction": "desc" } ],
    "search": ""
  }
}
```

### 17.7. Создание связи между сущностями
```http
POST /api/entities/{entityId}/relations
Content-Type: application/json

{
  "targetEntityId": 7,
  "relationKey": "",
  "relationType": "one_to_many",
  "nameJson": { "ru": "Заказы", "en": "Orders", "he": "הזמנות" },
  "inverseNameJson": { "ru": "Клиент", "en": "Customer", "he": "לקוח" }
}
```

### 17.8. Создание записи (данные) и смена статуса
```http
POST /api/entities/{entityId}/records
Content-Type: application/json

{
  "valuesJson": { "name": "ООО Ромашка", "phone": "+79990001122", "manager": 5 },
  "statusId": 11
}
```
- Чтение/фильтрация: `POST /api/entities/{entityId}/records/query` с телом `{ filters, filterConjunction, sorts, search, page, pageSize }`.
- Смена статуса = `PUT /api/records/{id}` с новым `statusId` (проходит проверку workflow).
- Архив/разархив: `POST /api/records/{id}/archive` / `POST /api/records/{id}/unarchive`.
- История изменений: `GET /api/records/{id}/audit`.

### 17.9. Создание роли с правами
```http
POST /api/roles
Content-Type: application/json

{
  "nameJson": { "ru": "Менеджер", "en": "Manager", "he": "מנהל" },
  "permissionsJson": {
    "superAdmin": false,
    "admin": { "pages": false, "entities": false, "roles": false, "users": false,
               "translations": false, "events": false, "modules": false,
               "googleDrive": false, "settings": false },
    "pageIds": [42],
    "records": { "12": { "view": true, "create": true, "update": true, "delete": false,
                          "scope": "own", "scopeFieldKeys": ["manager"] } }
  }
}
```

### 17.10. Создание пользователя и гостевой ссылки
```http
POST /api/users
{ "email": "u@example.com", "firstName": "Иван", "lastName": "Петров",
  "roleId": 3, "language": "ru", "password": "<пароль>", "isActive": true }
```
Беспарольный (гостевой) пользователь — без `password`; затем `POST /api/users/{id}/guest-links` создаёт шарящую ссылку (read-only JWT).

## 18. Рекомендуемый порядок настройки «с нуля» (для наполнения реальными данными)

1. **Брендинг** (`/settings`): имя приложения, подзаголовок, логотип, символ валюты.
2. **Страницы** (`/admin/pages`): спроектировать дерево меню (разделы, вложенность). Решить, какие страницы — обычные (под сущность), какие — дашборды, какие — зеркальные.
3. **Сущности** (`/admin/entities`): создать бизнес-объекты и привязать к страницам (одна сущность ↔ одна страница).
4. **Поля** для каждой сущности: выбрать типы, обязательность, опции `select`, фильтруемость, отображение в таблице, условное форматирование, итоги по числовым колонкам.
5. **Статусы** (если нужен жизненный цикл): default-статус, финальные, триггеры авто-архивации.
6. **Workflow** (опц.): переходы статусов с ролями, обязательными полями и действиями `set_field`.
7. **Связи**: типизированные связи между сущностями (учесть, что тип нельзя менять при наличии связей).
8. **Роли и права** (`/admin/roles`): какие страницы видны, CRUD по сущностям, scope `own/all` (+ `scopeFieldKeys`), права на поля (hidden/view/edit).
9. **Пользователи** (`/admin/users`): завести учётки, назначить роли, при необходимости — гостевые ссылки.
10. **Представления**: сохранённые фильтры/сортировки на сущность (видимость колонок — отдельно, через `showInTable`).
11. **Дашборды**: на страницах-дашбордах добавить виджеты (метрики/графики/таблицы/заметки) с видимостью по ролям.
12. **Модули** (`/admin/modules`): при необходимости включить Google Drive (через кнопку «Настройки» модуля) для файловых полей.
13. **Переводы** (`/admin/translations`): при необходимости донастроить строки UI на ru/en/he.
14. **Данные**: завести записи (через таблицу с инлайн-редактированием или record-эндпоинты).

## 19. Подсказки для ИИ при формировании задач на настройку

- Всегда указывайте `nameJson` тремя языками `{ru,en,he}`; недостающие языки имеют fallback ru → en → he.
- Системный ключ можно не задавать — он сгенерируется (но если важна стабильность ключа для интеграций — задайте явно: `[a-z][a-z0-9_]*`).
- Привязка: сначала страница, затем сущность с `pageId`. Помните про взаимоисключение «дашборд / зеркало / привязанная сущность».
- Для прав: `view` обязателен для любых write; для «только своих записей» используйте `scope:"own"` и заполните `scopeFieldKeys` полями типа `user` — иначе пользователь не увидит ничего.
- Для workflow: пока у сущности нет переходов — статусы свободно меняются; добавление первого перехода включает ограничения. `superAdmin` всегда в обход.
- Скрытые поля действительно скрыты на сервере (включая ответы и фильтры) — не полагайтесь на «спрятать в UI».
- Денежные значения форматируются символом валюты из `app_settings.currencySymbol`.
- Массового импорта данных нет — планируйте наполнение через создание записей (по одной/через UI) или через скрипт сидинга на стороне БД при первичной загрузке.
