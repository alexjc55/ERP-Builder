# Production ERP Builder

## Master Product Specification & Architecture Prompt

### Важно

Перед началом разработки необходимо внимательно прочитать весь документ.

Данный проект НЕ является CRM.

Данный проект НЕ является ERP для конкретного бизнеса.

Необходимо разработать универсальную API-First платформу-конструктор, позволяющую через интерфейс создавать производственные ERP-системы без изменения исходного кода.

Платформа должна позволять администратору создавать страницы, таблицы, поля, права доступа, workflow и бизнес-процессы через интерфейс.

Первая конфигурация платформы будет использоваться для управления производственными проектами, которые сейчас работают на Google Sheets.

---

# Главная идея платформы

Платформа должна работать по принципу:

Airtable + SmartSuite + Notion Database

но с акцентом на:

* производственные процессы
* сложные права доступа
* workflow
* большие таблицы
* API-интеграции

После завершения разработки любой новый бизнес-процесс должен настраиваться через интерфейс, а не через изменение кода.

---

# Критически важный принцип №1

## Metadata Driven Architecture

Платформа должна строиться на метаданных.

Любой элемент системы должен храниться в базе данных.

Например:

Страницы

Меню

Роли

Права

Сущности

Поля

Статусы

Workflow

Фильтры

Представления

должны храниться в PostgreSQL.

---

# Критически важный принцип №2

## Запрет на Hardcoded данные

Запрещено захардкодивать:

* страницы
* меню
* роли
* статусы
* названия колонок
* названия сущностей
* права доступа
* workflow
* бизнес-процессы

Плохо:

```javascript
const roles = ["Admin", "Manager"];
```

Плохо:

```javascript
<Menu>
  <Projects />
</Menu>
```

Плохо:

```javascript
const statuses = ["New", "Done"];
```

Все должно приходить из базы данных.

---

# Критически важный принцип №3

## API First

Любое действие выполняется через API.

Архитектура:

Frontend
↓
API
↓
Business Layer
↓
PostgreSQL

Frontend не содержит бизнес-логику.

Frontend является клиентом API.

В будущем через тот же API должны работать:

* Make
* AI Agents
* WhatsApp
* Telegram
* мобильные приложения
* внешние интеграции

---

# Критически важный принцип №4

## Database Driven UI

Интерфейс должен строиться по данным из базы.

При открытии системы:

1. Получить доступные страницы.
2. Построить меню.
3. Получить представления.
4. Получить поля.
5. Построить интерфейс.

Меню не должно существовать в коде.

Колонки не должны существовать в коде.

Названия страниц не должны существовать в коде.

---

# Пользовательские роли

Стартовые роли:

* Администратор
* Управляющий проектами
* Менеджер
* Производитель
* Финансовый директор

Это не фиксированный список.

Роли должны храниться в базе данных.

# Авторизация

Пользователь открывает страницу входа.

Вход:

* Email
* Пароль

После успешной авторизации:

Получить:

* роль
* права
* настройки пользователя

Система должна поддерживать:

* восстановление пароля
* смену пароля
* блокировку пользователя
* журнал входов

Самостоятельная регистрация отсутствует.

Пользователей создает только администратор.

# Мультиязычность

Платформа должна быть полностью готова к мультиязычности.

Важно:

Запрещено использовать:

* locale files
* translation files
* i18next json
* yaml translation files

---

Все переводы должны храниться в PostgreSQL.

Пример:

{
"ru":"Проекты",
"en":"Projects",
"he":"פרויקטים"
}

---

Любой текст метаданных должен храниться как JSONB.

Например:

Название поля.

Название статуса.

Описание.

Подсказка.

---

Добавление нового языка не должно требовать изменения структуры БД.

# Права доступа

Система должна иметь три независимых уровня безопасности.

---

## Уровень 1

Доступ к странице.

Для каждой роли:

Нет доступа

Просмотр

Создание

Редактирование

Удаление

---

## Уровень 2

Отображение записей.

Пример:

Показывать только записи:

Owner = CurrentUser

или

Manufacturer = CurrentUser

или

любое другое условие.

---

## Уровень 3

Права на колонки.

Для каждой роли:

Видеть

Редактировать

Скрыть

---

Пример:

Производитель видит:

Изделие

Количество

Статус

Чертеж

---

Производитель не видит:

Цена клиента

Прибыль

Финансовые показатели

---

# Workflow Engine

Любая сущность может иметь Workflow.

Workflow состоит из статусов.

Пример:

Создано

Подготовлено

Передано производителю

В производстве

Готово

Завершено

Архив

---

Для каждого перехода настраивается:

Кто может выполнить.

Какие поля обязательны.

Какие действия выполнить.

---

Пример:

Передача производителю.

Обязательные поля:

Производитель

Цена производства

Чертеж

Срок

---

После перехода:

Запись становится доступной производителю.

---

# Архивирование

Архив не является отдельной таблицей.

Записи не перемещаются.

Запись получает признак архивации.

---

Пример правила:

Если статус = Завершено

и прошло 30 дней

то:

Archived = true

---

Архивные записи скрываются из обычных представлений.

---

# История изменений

Для каждой записи хранить:

Пользователь

Дата

Поле

Старое значение

Новое значение

---

История должна работать для всех сущностей платформы.

# Главная цель разработки

После завершения разработки должна получиться универсальная metadata-driven платформа, на которой можно создавать производственные ERP-системы через интерфейс администратора без изменения исходного кода.

Первая конфигурация платформы должна заменить текущую систему управления производственными проектами, работающую на Google Sheets.
